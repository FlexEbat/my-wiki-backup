# 📚 Подробный конспект по Git

## 1. Базовые понятия
Прежде чем учить команды, важно понять, как Git мыслит:
*   **Репозиторий (Repository / Repo):** Папка с вашим проектом, которую отслеживает Git. В ней есть скрытая папка `.git`, где хранится вся история.
*   **Коммит (Commit):** «Снимок» (snapshot) состояния вашего кода в определенный момент времени. У каждого коммита есть уникальный хэш (например, `a1b2c3d`) и автор.
*   **Ветка (Branch):** Независимая линия разработки. Позволяет работать над новой фичей, не ломая основной код (обычно это ветка `main` или `master`).
*   **Удаленный репозиторий (Remote):** Версия вашего проекта, хранящаяся в интернете (на GitHub, GitLab, Bitbucket и т.д.).

### Три состояния файлов в Git:
1.  **Modified (Изменен):** Вы поменяли файл, но еще не сказали Git его сохранить.
2.  **Staged (В индексе / Подготовлен):** Вы отметили измененный файл для включения в следующий коммит.
3.  **Committed (Зафиксирован):** Данные надежно сохранены в локальной базе данных Git.

## 2. Первоначальная настройка
После установки Git нужно представиться системе, чтобы коммиты подписывались вашим именем.

```bash
git config --global user.name "Имя Фамилия"
git config --global user.email "ваш@email.com"
```
*Проверить настройки:* `git config --list`

---

## 3. Создание репозитория
**Вариант А: Начать новый проект с нуля**
Перейдите в папку с проектом и выполните:
```bash
git init
```
*(Это создаст скрытую папку `.git`)*

**Вариант Б: Скачать существующий проект из интернета**
```bash
git clone <URL_репозитория>
```
*(Например: `git clone https://github.com/user/project.git`)*

---

## 4. Основной рабочий процесс (Повседневные команды)

Это цикл, который вы будете повторять десятки раз в день.

**1. Проверка состояния:**
```bash
git status
```
*(Самая важная команда. Покажет, какие файлы изменены, какие добавлены в индекс, а какие Git вообще не отслеживает).*

**2. Добавление файлов в индекс (Staging):**
```bash
git add file.txt       # Добавить конкретный файл
git add .              # Добавить ВСЕ измененные и новые файлы в текущей папке
```

**3. Создание коммита:**
```bash
git commit -m "Добавил форму регистрации"
```
*(Флаг `-m` означает message. Пишите понятные сообщения!)*

**4. Просмотр истории:**
```bash
git log                # Полная история коммитов
git log --oneline      # Краткая история (каждый коммит в одну строку)
```

## 5. Ветвление (Branches)
Ветки нужны для безопасной разработки нового функционала.

**Просмотр веток:**
```bash
git branch             # Показать локальные ветки (текущая отмечена *)
git branch -a          # Показать локальные и удаленные ветки
```

**Создание и переход:**
```bash
git branch feature-login          # Создать ветку 'feature-login'
git checkout feature-login        # Перейти в нее (старый способ)
git switch feature-login          # Перейти в нее (новый, более понятный способ)

git checkout -b feature-login     # Создать и СРАЗУ перейти (популярный шорткат)
```

**Слияние веток (Merge):**
Чтобы влить изменения из ветки `feature-login` в `main`:
1. Перейдите в ветку, В КОТОРУЮ хотите влить (в `main`): `git switch main`
2. Выполните слияние: `git merge feature-login`
3. Если больше не нужна: `git branch -d feature-login` (удаление ветки)

---

## 6. Работа с сетью (GitHub / GitLab)

**Связывание локального репозитория с удаленным:**
*(Нужно только если вы делали `git init`, а не `git clone`)*
```bash
git remote add origin <URL_репозитория>
```

**Отправка изменений на сервер (Push):**
```bash
git push origin main              # Отправить ветку main на сервер
git push -u origin feature-branch # Отправить новую ветку и связать их (флаг -u)
```

**Получение изменений с сервера (Pull / Fetch):**
```bash
git pull origin main    # Скачивает новые изменения и СРАЗУ сливает их с вашим кодом
git fetch               # Только скачивает информацию об изменениях (не меняя ваш код)
```

## 7. Отмена изменений (Как всё исправить)

Git хорош тем, что в нем почти невозможно потерять код безвозвратно.

**Отменить изменения в файле до коммита (вернуть как было):**
```bash
git restore <файл>      # Сбросит все ваши изменения в файле до состояния последнего коммита
```

**Убрать файл из индекса (сделать Unstage, если случайно сделали git add):**
```bash
git restore --staged <файл>
```

**Отменить последний коммит, оставив файлы измененными:**
```bash
git reset HEAD~1
```

**Отменить коммит полностью и удалить изменения (ОСТОРОЖНО):**
```bash
git reset --hard HEAD~1
```

**Создать новый коммит, который отменяет действия старого (Безопасный способ для публичных веток):**
```bash
git revert <хэш_коммита>
```

## 8. Продвинутые и полезные фичи

### Файл `.gitignore`
Создайте файл с названием `.gitignore` в корне проекта. Впишите туда файлы и папки, которые Git должен ИГНОРИРОВАТЬ (например, пароли, логи, папку `node_modules` и т.д.).
Пример содержимого:
```text
node_modules/
.env
*.log
```

### Временное сохранение (Stash)
Вы пишете код, но срочно нужно переключиться на другую ветку, а коммитить недоделку не хочется:
```bash
git stash               # Прячет ваши изменения во временное хранилище
git stash pop           # Достает изменения обратно и применяет к текущему коду
git stash list          # Посмотреть список спрятанного
```

### Перебазирование (Rebase)
Альтернатива `merge`. Переносит вашу ветку так, будто она была создана от самого свежего коммита `main`. Делает историю красивой и линейной в одну линию.
```bash
git rebase main
```
*Золотое правило Rebase:* **Никогда не делайте rebase публичных веток (которые уже отправлены на сервер)!** Только локальных.

## 9. Решение частых проблем (Troubleshooting)

Даже опытные разработчики иногда путаются в Git. Здесь собраны команды для ситуаций, когда что-то пошло не так.

### 9.1. Очистка проекта от "мусора" (Удаление неотслеживаемых файлов)
Если вы наплодили тестовых файлов или папок, которые Git не отслеживает (Untracked), и хотите быстро вернуть папку проекта к чистому состоянию, `git reset --hard` не поможет (он трогает только отслеживаемые файлы). Нужен `clean`.

```bash
git clean -n -d         # ПРОВЕРКА (Dry run): покажет, что именно будет удалено (безопасно)
git clean -f -d         # УДАЛЕНИЕ: жестко удалит все неотслеживаемые файлы (-f) и папки (-d). ОСТОРОЖНО!
```

### 9.2. Ошибка в последнем коммите (Забыл файл или опечатался в сообщении)
Вы сделали `git commit`, но забыли добавить один файл, или написали с ошибкой. Не нужно делать новый коммит «fix typo»!

1. Добавьте забытый файл (если он есть): `git add forgotten_file.txt`
2. Выполните слияние с предыдущим коммитом:
```bash
git commit --amend -m "Новое, правильное сообщение"
```
*(Если нужно просто добавить файл, не меняя сообщение, используйте: `git commit --amend --no-edit`)*
**Важно:** Не делайте `--amend` для коммитов, которые вы уже отправили на GitHub (сделали push).

### 9.3. Разрешение конфликтов (Merge Conflicts)
Вы делаете `git pull` или `git merge`, и Git выдает: `CONFLICT (content): Merge conflict in file.txt`. Это значит, вы и кто-то другой изменили одну и ту же строчку кода.

**Что делать:**
1. Введите `git status`, чтобы увидеть, какие файлы конфликтуют (они будут красными).
2. Откройте эти файлы в редакторе (VS Code отлично подсвечивает конфликты). Внутри вы увидите:
```text
<<<<<<< HEAD
Ваш код (то, что было у вас)
=======
Чужой код (то, что прилетело из другой ветки)
>>>>>>> feature-branch
```
3. Удалите эти страшные стрелочки (`<<<<<<<`, `=======`, `>>>>>>>`) и оставьте тот код, который должен работать (или объедините оба варианта руками).
4. Сохраните файл.
5. Скажите Git, что вы всё решили:
```bash
git add file.txt
git commit -m "Resolved merge conflict"
```

### 9.4. Машина времени: как спасти "удаленный" код (Reflog)
Допустим, вы случайно сделали `git reset --hard` или удалили нужную ветку, и кажется, что код потерян навсегда. Git ведет скрытый журнал вообще ВСЕХ ваших действий.

```bash
git reflog
```
Вы увидите список всех ваших перемещений с хэшами (например, `HEAD@{2}`). Найдите момент ДО того, как вы совершили ошибку, скопируйте хэш (например, `abc1234`) и вернитесь туда:
```bash
git reset --hard abc1234
```
*Reflog хранит данные около 30 дней, так что спасти можно почти всё.*

### 9.5. Файл добавлен в .gitignore, но Git всё равно его видит
Это классика. Если вы добавили файл в проект, сделали коммит, а ПОТОМ добавили его в `.gitignore`, Git не перестанет за ним следить. Файл нужно вычистить из кэша.

**Решение (очистит кэш и применит новые правила .gitignore):**
```bash
git rm -r --cached .    # Удаляет всё из кэша (ваши файлы на диске НЕ удалятся!)
git add .               # Добавляет всё обратно (уже с учетом .gitignore)
git commit -m "Fix .gitignore"
```
