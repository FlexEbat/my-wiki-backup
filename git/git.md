#### 1. Три состояния файлов в Git
Чтобы понимать Git, нужно знать, где находятся ваши файлы:
1.  **Working Directory (Рабочая директория)** — ваши реальные файлы на диске прямо сейчас. Вы их редактируете.
2.  **Staging Area / Index (Индекс)** — "зал ожидания". Сюда вы добавляете файлы (`git add`), которые хотите включить в следующий коммит.
3.  **Repository / HEAD (Репозиторий)** — база данных Git. Сюда файлы попадают после `git commit`.

#### 2. Первоначальная настройка (делается один раз)
*   `git config --global user.name "Твое Имя"` — задать имя.
*   `git config --global user.email "твой@email.com"` — задать почту.
*   `git config --list` — проверить текущие настройки.

#### 3. Создание репозитория
*   `git init` — превратить текущую папку в Git-репозиторий (создастся скрытая папка `.git`).
*   `git clone <url>` — скопировать удаленный репозиторий на свой компьютер.

#### 4. Сохранение изменений (Коммиты)
*   `git add <файл>` — добавить конкретный файл в индекс.
*   `git add .` — добавить все измененные и новые файлы в индекс.
*   `git commit -m "Сообщение"` — зафиксировать изменения из индекса.
*   `git commit -a -m "Сообщение"` — автоматически добавить в индекс измененные файлы и сделать коммит (не работает для *новых* файлов).
*   `git commit --amend` — добавить изменения в *последний* коммит или изменить его сообщение.

#### 5. История и просмотр
*   `git log` — полная история коммитов.
*   `git log --oneline` — краткая история (каждый коммит в одну строку).
*   `git diff` — посмотреть, что именно изменилось в файлах (до `git add`).
*   `git diff --staged` — посмотреть, что изменилось в файлах, которые уже добавлены в индекс.

#### 6. Ветвление (Branches)
Ветки нужны для параллельной разработки (создания фич, исправления багов) без влияния на рабочий код.
*   `git branch` — список веток (звездочкой отмечена текущая).
*   `git branch <имя>` — создать новую ветку, но не переключаться на нее.
*   `git switch <имя>` *(или `git checkout`)* — переключиться на ветку.
*   `git branch -d <имя>` — удалить ветку (безопасно, не удалит, если изменения не влиты).
*   `git branch -D <имя>` — принудительно удалить ветку.

#### 7. Слияние (Merge)
*   `git merge <имя_ветки>` — влить указанную ветку в *текущую*.
    *   *Конфликт слияния (Merge Conflict):* Возникает, если один и тот же кусок кода был изменен в обеих ветках по-разному. Git попросит вас вручную отредактировать файл, оставить нужный код, а затем сделать `git add` и `git commit`.

#### 8. Работа с удаленными репозиториями (Remote)
Удаленный репозиторий — это версия вашего проекта, хранящаяся в интернете или локальной сети (например, на GitHub/GitLab). Вы можете подключать несколько таких репозиториев к одному локальному проекту.
*   **Добавление и управление адресами:**
    *   `git remote add origin <url>` — привязать локальный репозиторий к удаленному (по умолчанию ваш главный удаленный сервер называют `origin`).
    *   `git remote add <имя> <url>` — добавить еще один удаленный репозиторий (часто используют имя `upstream` для связи с оригинальным проектом, если вы сделали его форк).
    *   `git remote -v` — посмотреть список всех привязанных серверов и их адреса.
    *   `git remote set-url origin <новый_url>` — **изменить** адрес существующего удаленного репозитория (полезно, если проект переехал на другой сервер или вы поменяли протокол с HTTPS на SSH).
    *   `git remote remove <имя>` *(или `git remote rm`)* — удалить привязку к удаленному репозиторию (сами файлы на сервере не удалятся, только связь с ними).
    *   `git remote rename <старое_имя> <новое_имя>` — переименовать привязку (например, из `origin` в `my-server`).

*   **Обмен данными с сервером:**
    *   `git fetch` — скачать информацию об изменениях с сервера, но **не применять** их к вашим файлам.
    *   `git pull` — скачать изменения с сервера и сразу влить их в вашу текущую ветку (по сути это `git fetch` + `git merge`).
    *   `git push` — отправить ваши коммиты на сервер.
    *   `git push -u origin <имя_ветки>` — отправить новую локальную ветку на сервер в первый раз и сказать Git в будущем связывать её с сервером просто по команде `git push`.
    *   `git push -d origin <имя_ветки>` — удалить ветку на удаленном сервере.

#### 9. Отмена изменений (ВАЖНО!)
*   **Как отменить изменения в файле до `git add`:**
    *   `git restore <файл>` *(или `git checkout -- <файл>`)* — вернет файл к состоянию последнего коммита (изменения будут потеряны!).
*   **Как убрать файл из индекса (сделали `git add`, но передумали):**
    *   `git restore --staged <файл>` *(или `git reset HEAD <файл>`)*.
*   **Как отменить коммит:**
    *   `git revert <хэш_коммита>` — создает **новый** коммит, который отменяет изменения указанного коммита (самый безопасный способ для публичных веток).
    *   `git reset --soft HEAD~1` — удаляет последний коммит, но оставляет файлы в индексе (можно перекоммитить).
    *   `git reset --hard HEAD~1` — 🚨 **ОПАСНО!** Удаляет последний коммит и **все изменения в файлах**.

#### 10. Продвинутые и полезные фишки
*   **Прятки (Stash):** Если вы не доделали работу, но нужно срочно переключиться на другую ветку, коммит делать не хочется.
    *   `git stash` — прячет текущие изменения в "карман".
    *   `git stash pop` — достает изменения из "кармана" и применяет к текущей ветке.
    *   `git stash list` — посмотреть список спрятанного.
*   **.gitignore:** Файл в корне проекта. В нем прописывают файлы и папки, которые Git должен игнорировать (например, `node_modules`, скомпилированные файлы, пароли).
*   **Перебазирование (Rebase):** `git rebase <имя_ветки>` — переносит ваши текущие коммиты поверх указанной ветки. Делает историю чище (в линию), но **нельзя делать rebase публичных веток**, с которыми работают другие люди.
*   **Черри-пик (Cherry-pick):** `git cherry-pick <хэш_коммита>` — берет конкретный коммит из любой ветки и применяет его к вашей текущей ветке.
