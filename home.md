# 🐳 Подробный конспект по Docker и Docker Compose

## 1. Базовые понятия
Docker решает проблему «А у меня на компьютере это работает!». Он упаковывает ваше приложение и все его зависимости в изолированную среду.

*   **Образ (Image):** «Шаблон» или чертеж вашего приложения. Он неизменяемый (read-only). Содержит ОС, библиотеки и ваш код.
*   **Контейнер (Container):** Запущенный экземпляр образа. Вы можете запустить 10 контейнеров из одного образа. Если контейнер удалить, все данные внутри него сотрутся.
*   **Dockerfile:** Текстовый файл с инструкциями, как собрать ваш Образ (Image).
*   **Том (Volume):** Специальная папка, которая хранится на вашем жестком диске, но подключена к контейнеру. Нужна, чтобы данные (например, база данных) не удалялись при остановке контейнера.
*   **Registry (Реестр):** Облако, где хранятся образы (самое популярное — Docker Hub). Аналог GitHub для Docker.

## 2. Повседневные команды Docker

### Работа с Образами (Images)
```bash
docker pull ubuntu         # Скачать готовый образ из Docker Hub
docker images              # Показать все образы на вашем ПК
docker rmi <id_образа>     # Удалить образ (rm image)
```

### Работа с Контейнерами (Containers)
**Запуск контейнера:**
```bash
docker run -d -p 8080:80 --name my_server nginx
```
*Разбор флагов:*
*   `-d` (detach): запустить в фоновом режиме (чтобы терминал не завис).
*   `-p 8080:80` (port): проброс портов. Порт `8080` на вашем ПК связывается с портом `80` внутри контейнера.
*   `--name`: дать контейнеру понятное имя (иначе Docker придумает его сам).

**Управление контейнерами:**
```bash
docker ps                  # Показать ЗАПУЩЕННЫЕ контейнеры
docker ps -a               # Показать ВООБЩЕ ВСЕ (включая остановленные)
docker stop my_server      # Остановить контейнер
docker start my_server     # Запустить снова
docker rm my_server        # Удалить контейнер (он должен быть остановлен)
```

## 3. Написание Dockerfile
Чтобы упаковать **свое** приложение, нужно создать файл с именем `Dockerfile` (без расширения) в корне проекта.

Пример для Node.js приложения:
```dockerfile
# 1. Берем базовый образ
FROM node:18-alpine

# 2. Указываем рабочую папку внутри контейнера
WORKDIR /app

# 3. Копируем файлы package.json и устанавливаем зависимости
COPY package*.json ./
RUN npm install

# 4. Копируем весь остальной код
COPY . .

# 5. Указываем порт, который слушает приложение
EXPOSE 3000

# 6. Команда для запуска приложения
CMD ["npm", "start"]
```
**Сборка образа из этого файла:**
```bash
docker build -t my-app-name .  # -t задает имя (тег), точка в конце означает "в текущей папке"
```

## 4. Docker Compose
Если у вас есть приложение + база данных + кэш (Redis), запускать их по отдельности через `docker run` — мучение. **Docker Compose** позволяет описать всю инфраструктуру в одном файле `docker-compose.yml` и запустить одной командой.

Пример `docker-compose.yml`:
```yaml
version: '3.8' # Версия синтаксиса

services:
  web:
    build: .             # Собрать образ из локального Dockerfile
    ports:
      - "3000:3000"      # Проброс портов
    depends_on:
      - db               # Запускать только после базы данных

  db:
    image: postgres:15   # Взять готовый образ из интернета
    environment:         # Переменные окружения (пароли, логины)
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    volumes:
      - pgdata:/var/lib/postgresql/data # Подключение тома для сохранения БД

volumes:
  pgdata: # Объявление тома
```

**Команды Docker Compose:**
```bash
docker compose up -d       # Поднять весь проект (создать и запустить всё в фоне)
docker compose down        # Остановить и удалить все контейнеры проекта
docker compose ps          # Посмотреть статус сервисов
docker compose logs -f     # Смотреть логи всех сервисов в реальном времени
```

## 5. Решение частых проблем (Troubleshooting & Очистка)

Docker любит «съедать» место на жестком диске и оставлять мусор. Вот команды для спасения.

### 5.1. Тотальная очистка (Аналог git clean -fd)
Если Docker занимает 50 ГБ на диске, или вы запутались в старых образах и хотите начать с чистого листа:

```bash
# Удалить все остановленные контейнеры, неиспользуемые сети, образы без тегов ("dangling"):
docker system prune

# ТОТАЛЬНАЯ ЗАЧИСТКА: Удалить ВООБЩЕ ВСЁ, что сейчас не запущено (включая скачанные образы):
docker system prune -a

# ТОТАЛЬНАЯ ЗАЧИСТКА + УДАЛЕНИЕ ТОМОВ (БАЗ ДАННЫХ):
# ОСТОРОЖНО! Это удалит ваши локальные базы данных в Docker!
docker system prune -a --volumes
```

### 5.2. Убить все контейнеры (Экстренная остановка)
Если запущена куча всего, порты заняты, и вы не хотите разбираться, кто есть кто:
```bash
docker rm -f $(docker ps -aq)
# Эта команда берет ID всех существующих контейнеров (-aq) и жестко удаляет их (-f).
```

### 5.3. Ошибка "Port is already allocated" (Порт занят)
Вы пытаетесь запустить контейнер, а Docker говорит: `Bind for 0.0.0.0:8080 failed: port is already allocated`.
*   **Причина:** Какой-то другой контейнер (или программа на вашем ПК) уже использует этот порт.
*   **Решение 1:** Найти, кто занял порт (`docker ps`), и остановить его (`docker stop <id>`).
*   **Решение 2:** В `docker run` или `docker-compose.yml` изменить левую часть порта (порт вашего ПК). Например, сделать `8081:8080`.

### 5.4. Контейнер падает сразу после запуска
Вы делаете `docker-compose up -d`, проверяете через `docker ps`, а вашего контейнера там нет (он в `docker ps -a` со статусом Exited).
**Что делать:** Обязательно смотрите логи!
```bash
docker logs <имя_контейнера>
# Если логов много, смотрите последние 50 строк:
docker logs --tail 50 <имя_контейнера>
```
*Обычно там будет понятная ошибка приложения (например, "Не могу подключиться к базе данных").*

### 5.5. Как "зайти" внутрь работающего контейнера
Если вам нужно проверить, создался ли файл внутри, или выполнить команду напрямую в Linux-среде контейнера:
```bash
docker exec -it <имя_контейнера> bash
# Если bash не работает (в образах alpine его нет), используйте sh:
docker exec -it <имя_контейнера> sh
```
*(Для выхода из контейнера напишите `exit`)*

## 6. Хорошие практики (Best Practices)

1. **Используйте `.dockerignore`.**
   Создайте этот файл рядом с `Dockerfile`. Впишите туда `node_modules`, `.git`, логи. Это ускорит сборку образа в 10 раз и сделает его легче, так как Docker не будет копировать этот мусор внутрь.
2. **Не используйте тег `latest` в production.**
   Пишите `FROM node:18.16.0` вместо `FROM node:latest` или `FROM node`. Иначе однажды скачается новая мажорная версия, и всё сломается.
3. **Один контейнер — один процесс.**
   Не пытайтесь запихнуть PHP, Nginx и MySQL в один `Dockerfile`. Сделайте 3 разных контейнера и свяжите их через Docker Compose. Это философия Docker.
4. **Порядок команд в Dockerfile имеет значение.**
   Docker кэширует каждый шаг. Поэтому `COPY package.json` и `RUN npm install` пишут **до** копирования всего остального кода (`COPY . .`). Если вы измените только логику кода (например `app.js`), Docker не будет заново скачивать все библиотеки (npm install), а возьмет их из кэша.
