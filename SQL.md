## 1. Многотабличные запросы и оператор JOIN
Используется для объединения данных из нескольких таблиц, связанных между собой (обычно через первичные и внешние ключи).
### Структура:
SQL
```
SELECT поля_таблиц
FROM таблица_1
[INNER] | [[LEFT | RIGHT | FULL][OUTER]] JOIN таблица_2
    ON условие_соединения;
```
### Виды соединений:
- **INNER JOIN**: Возвращает только те строки, для которых есть соответствие в обеих таблицах.
- **LEFT JOIN**: Возвращает все строки из левой таблицы и подходящие строки из правой. Если совпадений нет, подставляется `NULL`.
- **RIGHT JOIN**: Возвращает все строки из правой таблицы и подходящие из левой.
- **FULL JOIN**: Возвращает все записи из обеих таблиц, объединяя их по условию (некоторые СУБД требуют эмуляции через `UNION`).
**Пример:** Вывод имен членов семьи и их трат.
SQL
```
SELECT FamilyMembers.member_name, Payments.unit_price * Payments.amount AS total
FROM FamilyMembers
INNER JOIN Payments ON FamilyMembers.member_id = Payments.family_member;
```

## 2. Ограничение выборки (LIMIT)
Позволяет извлечь определенный диапазон записей. Полезно для пагинации (постраничного вывода).
### Структура:
```SQL
-- MySQL синтаксис
SELECT поля FROM таблица
LIMIT [количество_пропущенных,] количество_для_вывода;

-- Или с использованием OFFSET
SELECT поля FROM таблица
LIMIT количество_для_вывода OFFSET количество_пропущенных;
```
**Пример:** Получить две самые дешевые комнаты.
```SQL
SELECT * FROM Rooms
ORDER BY price ASC
LIMIT 2;
```

## 3. Подзапросы (Subqueries)
Подзапрос — это запрос внутри другого запроса.
### А. Скалярные подзапросы (1 строка, 1 столбец)
Используются с операторами сравнения (
**Пример:** Найти владельца самого дорогого жилья.
```sql
SELECT * FROM Users
WHERE id = (SELECT owner_id FROM Rooms ORDER BY price DESC LIMIT 1);
```
### Б. Столбцовые подзапросы (N строк, 1 столбец)
Используются с операторами `IN`, `ANY`, `ALL`.
- **IN**: Значение входит в список.
- **ANY**: Условие верно хотя бы для одного значения из списка.
- **ALL**: Условие верно для всех значений в списке.

**Пример:** Товары, которые ни разу не покупались.
```sql
SELECT good_name FROM Goods
WHERE good_id NOT IN (SELECT DISTINCT good FROM Payments);
```
### В. Строковые подзапросы (Сравнение кортежей)
Позволяют сравнивать сразу несколько колонок.
**Пример:** Найти комнаты с такими же удобствами, как у комнаты №11.

```sql
SELECT * FROM Rooms
WHERE (has_tv, has_internet) = (SELECT has_tv, has_internet FROM Rooms WHERE id = 11);
```

---
## 4. Коррелированные подзапросы
Подзапросы, которые ссылаются на колонки внешнего запроса. Выполняются по одному разу для каждой строки основного запроса.
**Пример:** Имя каждого члена семьи и его самая дорогая покупка.

```sql
SELECT member_name, 
    (SELECT MAX(unit_price) FROM Payments WHERE family_member = FamilyMembers.member_id) AS max_price
FROM FamilyMembers;
```

## 5. Объединение результатов (UNION)
Используется для склеивания результатов двух независимых запросов с одинаковой структурой столбцов.
### Структура:
```sql
SELECT поля FROM таблица_1
UNION [ALL]
SELECT поля FROM таблица_2;
```
- **UNION**: Убирает дубликаты.
- **UNION ALL**: Оставляет все строки, включая повторения.
- **INTERSECT**: Только общие строки для обоих запросов.
- **EXCEPT**: Строки из первого запроса, которых нет во втором.

## 6. Условная логика (IF, IFNULL)
Позволяет изменять выводимые значения в зависимости от условий прямо в `SELECT`.
### Структура IF:
`IF(условие, значение_если_истина, значение_если_ложь)`
**Пример:** Замена 1/0 на понятный текст.

```sql
SELECT id, IF(has_tv = 1, 'Есть', 'Нет') AS tv_status FROM Rooms;
```
### Структура IFNULL:
`IFNULL(проверяемое_поле, значение_замены_если_NULL)`
**Пример:** Замена отсутствующего отчества.

```sql
SELECT first_name, IFNULL(middle_name, 'Empty') FROM Teacher;
```

---
## 7. Обобщенные табличные выражения (CTE)
CTE (Common Table Expressions) — это временные результирующие наборы данных, которые можно использовать внутри одного запроса. Они делают код чище и позволяют строить рекурсию.
### Обычное CTE:

```sql
WITH название_cte [(столбец_1, столбец_2...)] AS (
    подзапрос
)
SELECT * FROM название_cte;
```
### Рекурсивное CTE:
Используется для работы с иерархиями (например, дерево начальников и подчиненных).

```sql
WITH RECURSIVE название_cte AS (
    -- 1. Начальный набор (базовый случай)
    SELECT ... FROM ... WHERE ...
    
    UNION ALL
    
    -- 2. Рекурсивный шаг (присоединение к самому себе)
    SELECT ... FROM таблица
    INNER JOIN название_cte ON условие_связи
)
SELECT * FROM название_cte;
```

# Конспект по SQL: Изменение данных (INSERT, UPDATE, DELETE)
## 1. Добавление данных (INSERT)
Используется для создания новых записей. Можно вставлять данные вручную или копировать их из других таблиц.
### Структура (Синтаксис VALUES):
```sql
INSERT INTO имя_таблицы (поле_1, поле_2, ...)
VALUES (значение_1, значение_2, ...),
       (значение_3, значение_4, ...); -- Можно вставить сразу несколько строк
```
### Структура (Синтаксис SELECT):
Используется для копирования данных или вставки результатов вычислений.
```sql
INSERT INTO имя_таблицы (поле_1, поле_2, ...)
SELECT поле_а, поле_б, ... FROM другая_таблица
[WHERE условие];
```
**Ключевые правила:**
- **Явное указание полей**: Всегда перечисляйте поля после `INSERT INTO`. Это защитит ваш код, если структура таблицы изменится (например, добавится новая колонка).
- **AUTO_INCREMENT**: Если для первичного ключа настроено автозаполнение, при вставке передавайте `NULL` или `0` — база данных сама подставит следующее число.
- **Соответствие**: Количество и порядок значений в `VALUES` должны строго совпадать со списком полей.

---
## 2. Обновление данных (UPDATE)
Используется для редактирования уже существующих строк.
### Структура:

```sql
UPDATE имя_таблицы
SET поле_1 = значение_1, 
    поле_2 = вычисляемое_значение
[WHERE условие_отбора];
```

**Важные нюансы:**
- **Опасность!** Если забыть блок `WHERE`, изменения применятся ко **всем** строкам таблицы. Это классическая ошибка, которая может привести к потере данных.
- **Вычисления**: В блоке `SET` можно использовать текущее значение поля: `SET price = price * 1.1` (поднять цену на 10%).
- **Присваивание**: Можно копировать значения из одного столбца в другой в рамках одной строки: `SET old_price = price, price = 100`.

---
## 3. Удаление данных (DELETE и TRUNCATE)
В SQL есть два способа «очистки», и они работают по-разному.
### Оператор DELETE (Хирургический инструмент)
Удаляет конкретные записи, подходящие под условие.

```sql
DELETE FROM имя_таблицы
[WHERE условие];
```
### Оператор TRUNCATE (Топор)
Полностью очищает таблицу. Работает быстрее, так как не логирует удаление каждой строки и просто пересоздает таблицу.

```sql
TRUNCATE TABLE имя_таблицы;
```

**Сравнение:**
### **Оператор DELETE** (Хирургическое удаление)
- **Условие (`WHERE`):** Позволяет удалять записи выборочно. Можно удалить одну строку, группу строк или вообще всё.
- **Скорость:** Работает медленнее, так как СУБД удаляет каждую строку по отдельности и записывает это в лог (журнал транзакций).
- **Счетчик ID (`AUTO_INCREMENT`):** Сохраняет текущее значение. Если вы удалите последнюю запись с ID 10, следующая новая запись получит ID 11.
- **Транзакции:** Безопасен. Если вы обернули запрос в транзакцию, удаление можно «откатить» командой `ROLLBACK`.
### **Оператор TRUNCATE** (Полная очистка)
- **Условие (`WHERE`):** Не поддерживает фильтрацию. Этот оператор всегда очищает таблицу полностью.
- **Скорость:** Работает очень быстро. Технически он не удаляет строки одну за другой, а просто «отрезает» данные или пересоздает пустую таблицу заново.
- **Счетчик ID (`AUTO_INCREMENT`):** Сбрасывает счетчик в начальное состояние. После очистки первая вставленная запись снова получит ID 1.
- **Транзакции:** В большинстве СУБД (например, в MySQL) выполняется мгновенно и не подлежит откату. Это считается операцией изменения структуры (DDL), а не данных.

Если нужно удалить **часть** данных — используй `DELETE`. Если нужно **полностью обнулить** таблицу и начать жизнь с чистого листа (с первого ID) — используй `TRUNCATE`.

---
## 4. Удаление с объединением (JOIN)
Если нужно удалить записи из одной таблицы на основе данных из другой.
### Структура:

```sql
DELETE имя_таблицы_1 
FROM имя_таблицы_1 
JOIN имя_таблицы_2 ON условие_связи
WHERE условие_фильтрации;
```
_Если нужно удалить данные из **обеих** таблиц сразу, перечислите их через запятую после `DELETE`: `DELETE t1, t2 FROM ...`._
**Пример:** Удалить все бронирования (`Reservations`) для комнат (`Rooms`), где нет кухни.

```sql
DELETE Reservations 
FROM Reservations 
JOIN Rooms ON Reservations.room_id = Rooms.id
WHERE Rooms.has_kitchen = false;
```

---

> Перед тем как выполнить `UPDATE` или `DELETE` с условием `WHERE`, сначала выполните этот же запрос как `SELECT *`. Если вы увидите в результатах именно те строки, которые планировали изменить — смело запускайте основную команду.

## 🔢 1. Числовой тип данных в SQL

Создание числовых данных в SQL достаточно простое: можно ввести число как литерал, получить его из столбца таблицы или сгенерировать путём вычислений.

При вычислениях можно использовать все стандартные арифметические операции (+, -, *, / и прочие) и изменять приоритеты вычислений с помощью скобок.

-- MySQL 8.1

```

| calc_example |  
| :--- |  
| 4 |

### Математические функции

Для большинства математических вычислений (степень, корень, логарифм) в SQL есть встроенные числовые функции.

|   |   |
|---|---|
|Имя функции|Описание|
|POW(num, power)|Вычисляет число в указанной степени|
|SQRT(num)|Вычисляет квадратный корень числа|
|LOG(base, num)|Вычисляет логарифм числа по указанному основанию|
|EXP(num)|Вычисляет экспоненту (e в степени num)|
|SIN(num)|Вычисляет синус числа|
|COS(num)|Вычисляет косинус числа|
|TAN(num)|Вычисляет тангенс числа|

> Список всех числовых функций, их описание и примеры можно посмотреть в официальном справочнике.

### Округление чисел

При работе с числами с плавающей точкой не всегда требуется полная точность. Например, денежные транзакции можно хранить с точностью до 6 знаков после запятой, а отображать — до 2 (до копеек).

Для округления предусмотрены 4 функции: CEILING, FLOOR, ROUND, TRUNCATE.

#### CEILING и FLOOR

Направлены на округление к ближайшему целому числу в большую (CEILING) и в меньшую (FLOOR) сторону.

code SQL

expand_less

```
SELECT CEILING(69.69) AS ceiling, FLOOR(69.69) AS floor;
```

|   |   |
|---|---|
|ceiling|floor|
|70|69|

#### ROUND

Округляет к ближайшему целому числу (десятичная часть >= 0.5 округляется в большую сторону, иначе в меньшую).

code SQL

expand_less

```
SELECT ROUND(69.499), ROUND(69.5), ROUND(69.501);
```

|   |   |   |
|---|---|---|
|ROUND(69.499)|ROUND(69.5)|ROUND(69.501)|
|69|70|70|

ROUND также может принимать **второй аргумент** — количество знаков после запятой.

```

|   |   |   |
|---|---|---|
|ROUND(69.7171, 1)|ROUND(69.7171, 2)|ROUND(69.7171, 3)|
|69.7|69.72|69.717|

|---|---|---|
|ROUND(1691.7, -1)|ROUND(1691.7, -2)|ROUND(1691.7, -3)|

#### TRUNCATE

Аналогична ROUND (принимает 2-й параметр), но **не округляет, а просто отбрасывает** ненужные цифры.

code SQL

|TRUNCATE(69.7979, 1)|TRUNCATE(69.7979, 2)|TRUNCATE(69.7979, 3)|

### Работа со знаковыми числами

При наличии отрицательных значений полезны функции SIGN и ABS.

- **SIGN()** — возвращает -1 (если число отрицательное), 0 (если нулевое) и 1 (если положительное).
    
- **ABS()** — возвращает абсолютное значение (модуль) числа.
    

code SQL

downloadcontent_copy

expand_less

```
SELECT SIGN(-69), SIGN(0), SIGN(69);
-- Результат: -1 | 0 | 1

SELECT ABS(-69), ABS(0), ABS(69);
-- Результат: 69 | 0 | 69
```

## 📅 2. Дата и время в SQL

Временны́е данные являются наиболее сложными 🤯. Причины:

1. Множество способов задания даты и времени.
    
2. Наличие временных зон.
    
3. Неочевидность вычислений (например, сложность вычисления точного возраста).
    

### Генерация временных данных

Получить дату можно 3 способами:

1. Скопировать из существующего столбца.
    
2. Задать через строковое представление.
    
3. Вызвать встроенные функции.
    

### Строковое представление временных данных

|   |   |
|---|---|
|Тип|Формат по умолчанию|
|**DATE**|YYYY-MM-DD|
|**DATETIME**|YYYY-MM-DD hh:mm:ss|
|**TIMESTAMP**|YYYY-MM-DD hh:mm:ss|
|**TIME**|hhh:mm:sss|
|**YEAR**|YYYY (полный формат)<br>YY или Y (сокращённый: 0-69 = 2000-2069; 70-99 = 1970-1999)|

Примечание: Допускается использовать любые знаки пунктуации как разделители, либо писать слитно.

code SQL

downloadcontent_copy

expand_less

```
SELECT  
    CAST("2022-06-16 16:37:23" AS DATETIME) AS datetime_1,
    CAST("2014/02/22 16*37*22" AS DATETIME) AS datetime_2,
    CAST("20220616163723" AS DATETIME) AS datetime_3,
    CAST("2021-02-12" AS DATE) AS date_1,
    CAST("160:23:13" AS TIME) AS time_1,
    CAST("89" AS YEAR) AS year;
```

### Функции генерации дат

Если строка не подходит под стандартные форматы, используйте STR_TO_DATE:

-- Результат: 1998-11-13T00:00:00.000Z
```

Для получения **текущей** даты/времени:

```

### Функции извлечения временных данных

|   |   |
|---|---|
|Функция|Описание|
|YEAR|Возвращает год для указанной даты|
|MONTH|Возвращает числовое значение месяца (1-12)|
|DAY|Возвращает день месяца (1-31)|
|HOUR|Возвращает значение часа (0-23)|
|MINUTE|Возвращает значение минут (0-59)|

### Отличие DATETIME от TIMESTAMP

|   |   |   |
|---|---|---|
|Критерий|DATETIME|TIMESTAMP|
|**Диапазон**|от 1000-01-01 00:00:00 до 9999-12-31 23:59:59|от 1970-01-01 00:00:00 до 2038-01-19 03:14:07|
|**Часовой пояс**|Не учитывается. Отображается как было сохранено.|Учитывается. Отображается с учётом часового пояса сервера.|

#### Часовые пояса

Мир разделён на 24 часовых пояса (отсчёт от UTC). Настройка в MySQL:

code SQL

expand_less

```
SET GLOBAL time_zone = '+03:00';    -- глобально
SET time_zone = '+03:00';           -- для текущего пользователя
SET @@session.time_zone = '+03:00'; -- для текущей сессии
```

### Примеры задач: Определение возраста

❌ **Плохой способ (Разница годов):**

code SQL

downloadcontent_copy

expand_less

```
SELECT YEAR(NOW()) - YEAR('2003-07-03 14:10:26');
```

Проблема: Не учитывает, был ли уже день рождения в этом году.

❌ **Средний способ (Разница в днях / 365):**

code SQL

downloadcontent_copy

expand_less

```
SELECT FLOOR(DATEDIFF(NOW(), '2003-07-03 14:10:26') / 365);
```

Проблема: Не учитывает високосные года (искажает статистику).

✅ **Идеальный способ (TIMESTAMPDIFF):**

code SQL

downloadcontent_copy

expand_less

```
SELECT TIMESTAMPDIFF(YEAR, '2003-07-03 14:10:26', NOW());
```

## 🔄 3. Функции преобразования типов (CAST и CONVERT)

СУБД часто делает неявное преобразование (например, строку в число):

code SQL

downloadcontent_copy

expand_less

```
SELECT '50' > 49 AS comparison_1, '50' > 51 AS comparison_2;
-- Результат: comparison_1 = 1 (True), comparison_2 = 0 (False)
```

Для **явного** преобразования используются CAST и CONVERT.  
**Синтаксис:**

code SQL

downloadcontent_copy

expand_less

```
CAST(значение AS тип_для_конвертации);
CONVERT(значение, тип_для_конвертации);
```

**Пример:**

code SQL

downloadcontent_copy

expand_less

```
SELECT CAST(12005.6 AS DECIMAL) AS cast_example, 
       CONVERT(12005.4, DECIMAL) AS convert_example;
```

|   |   |

|cast_example|convert_example|
|12006|12005|

### Доступные типы для конвертации:

- DATE, DATETIME, TIME, YEAR
    
- DECIMAL[(M[,D])] (по умолчанию D=0, M=10)
    
- CHAR[(N)]
    
- SIGNED, UNSIGNED (BIGINT)
    
- BINARY

## 🪟 4. Оконные функции SQL

Оконные функции — мощный инструмент, позволяющий проводить сложные вычисления по группам строк, связанным с текущей строкой.

### Принцип работы

В отличие от обычных агрегатных функций (которые схлопывают строки), оконные функции сегментируют запрос на "окна", вычисляют агрегат, но **оставляют исходные строки на месте**, добавляя результат в новый столбец.

Окно может быть:

1. Всей таблицей.
    
2. Партициями (группами строк по полю).
    
3. "Скользящим" диапазоном (например, текущая + предыдущая строка).
    

### Синтаксис оконной функции

code SQL

downloadcontent_copy

expand_less

```
SELECT <оконная_функция>(<поле_таблицы>)
OVER (
      [PARTITION BY <столбцы_для_разделения>][ORDER BY <столбцы_для_сортировки>][ROWS|RANGE <определение_диапазона_строк>]
)
```

- PARTITION BY — делит на изолированные группы.
    
- ORDER BY — сортирует внутри окна.
    
- ROWS|RANGE — ограничивает рамки окна до/после текущей строки.
    

### Пример: Подсчёт учеников в классе

Сначала мы джоиним таблицы, чтобы получить класс ученика, а затем применяем оконную функцию COUNT(*) OVER(PARTITION BY class).

code SQL

downloadcontent_copy

expand_less

```
SELECT
    Student.first_name,
    Student.last_name,
    Student_in_class.class,
    COUNT(*) OVER (PARTITION BY Student_in_class.class) AS student_count_in_class
FROM Student_in_class
JOIN Student ON Student_in_class.student = Student.id;
```

**Часть результата:**  
| first_name | last_name | class | student_count_in_class |  
| :--- | :--- | :--- | :--- |  
| Egor | Belyakov | 1 | 7 |  
| Artyom | Nikitin | 1 | 7 |  
| ... | ... | ... | ... |  
| Kirill | SHubin | 3 | 8 |

Здесь для каждого ученика вывелось общее количество детей в его классе (7 для 1-го класса, 8 для 3-го и т.д.).

> **Порядок выполнения:** Оконные функции отрабатывают предпоследним шагом — после WHERE, GROUP BY и HAVING, но перед финальным ORDER BY.

## 📉 6. Сортировка внутри окна (ORDER BY)

Сортировка внутри окна (ORDER BY в блоке OVER) позволяет ранжировать данные или считать показатели с накоплением (нарастающим итогом).

### Пример: Накопление суммы трат пользователя

Если мы просто используем SUM() OVER(PARTITION BY user_id), мы получим **финальную** сумму для всех строк пользователя.

Чтобы увидеть, **как росли траты со временем**, добавляем сортировку:

code SQL

downloadcontent_copy

expand_less

```
SELECT user_id, start_date, total AS reservation_price,
       SUM(total) OVER (
           PARTITION BY user_id
           ORDER BY start_date
       ) AS cumulative_total
FROM Reservations;
```

**Особенность:** При добавлении ORDER BY без указания рамок, SQL по умолчанию применяет правило RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (от начала партиции до текущей строки). Именно поэтому сумма накапливается!

## 🖼️ 7. Рамки окон (ROWS / RANGE)

- **Партиция** делит всю таблицу на группы.
    
- **Окно** (рамки) определяет, какие именно строки внутри партиции брать для текущей вычисляемой строки.
    

### Синтаксис

code SQL

downloadcontent_copy

expand_less

```
ROWS|RANGE BETWEEN <начало границы окна> AND <конец границы окна>
```

**Возможные границы:**

- UNBOUNDED PRECEDING — все предшествующие строки.
    
- N PRECEDING — N строк до текущей.
    
- CURRENT ROW — текущая строка.
    
- N FOLLOWING — N строк после текущей.
    
- UNBOUNDED FOLLOWING — все последующие строки.
    

**Разница между ROWS и RANGE:**

- **ROWS** — физические строки. Считает строки буквально (например, ровно 1 строка назад).
    
- **RANGE** — логические значения. Группирует строки с одинаковыми значениями в ORDER BY (например, все строки с одной и той же датой попадут в окно одновременно).

## 🛠️ 8. Основные оконные функции

Оконные функции делятся на 3 группы:

### 1. Агрегатные

SUM, COUNT, AVG, MAX, MIN.

code SQL

downloadcontent_copy

expand_less

```
SELECT id, home_type, price,
	SUM(price) OVER(PARTITION BY home_type) AS "Sum",
	MAX(price) OVER(PARTITION BY home_type) AS "Max"
FROM Rooms;
```

### 2. Ранжирующие

Обязательно требуют наличия ORDER BY в конструкции OVER.

- **ROW_NUMBER()** — уникальная нумерация строк (1, 2, 3, 4...).
    
- **RANK()** — ранг строки с пропусками (если есть дубли, ранги будут: 1, 2, 2, 4).
    
- **DENSE_RANK()** — ранг строки без пропусков (1, 2, 2, 3).
    

code SQL

downloadcontent_copy

expand_less

```
SELECT id, home_type, price,
	ROW_NUMBER() OVER(PARTITION BY home_type ORDER BY price) AS "row_number",
	RANK() OVER(PARTITION BY home_type ORDER BY price) AS "rank",
	DENSE_RANK() OVER(PARTITION BY home_type ORDER BY price) AS "dense_rank"
FROM Rooms;
```

### 3. Функции смещения

Позволяют обращаться к другим строкам окна:

- **LAG(col, N, default)** — значение из строки на N строк выше.
    
- **LEAD(col, N, default)** — значение из строки на N строк ниже.
    
- **FIRST_VALUE(col)** — первое значение в окне.
    
- **LAST_VALUE(col)** — последнее значение в окне.
    

code SQL

downloadcontent_copy

expand_less

```
SELECT id, home_type, price,
	LAG(price) OVER(PARTITION BY home_type ORDER BY price) AS "lag",
	LEAD(price) OVER(PARTITION BY home_type ORDER BY price) AS "lead",
	FIRST_VALUE(price) OVER(PARTITION BY home_type ORDER BY price) AS "first_value"
FROM Rooms;
```

## 🏦 9. Транзакции

Если бы приложения и серверы работали 100% времени без сбоев, проблемы одновременного доступа не существовало бы. Но в реальности это не так.

**Транзакция** — это последовательность операций с базой данных, которые выполняются как единое целое ("Всё или ничего"). Гарантирует, что либо все SQL-инструкции выполнятся успешно, либо ни одна не будет применена.

## 🔒 10. Блокировки в СУБД

Когда тысячи пользователей меняют данные одновременно, СУБД должна гарантировать целостность данных с помощью блокировок.

**Блокировка** — это метод ограничения доступа к данным, пока одна транзакция с ними работает.

### Гранулярность блокировок

1. **Блокировка таблиц:** Блокирует всю таблицу. Быстро, но вызывает долгие ожидания для других пользователей.
    
2. **Блокировка страниц:** Блокирует сегмент памяти (2-16 Кбайт).
    
3. **Блокировка строк:** Блокирует только конкретную строку. Позволяет пользователям изменять разные строки одной таблицы одновременно. (По умолчанию движок InnoDB в MySQL использует именно блокировку строк).

## 🚀 11. Создание транзакций

Рассмотрим пример перевода $1000 с одного счёта на другой.  
Если деньги списались, но сервер упал до зачисления — деньги исчезнут. Транзакции решают эту проблему.

code SQL

downloadcontent_copy

expand_less

```
-- Начало транзакции
START TRANSACTION;

-- Проверка баланса (псевдокод логики)
SELECT @balance := user_balance FROM accounts WHERE user_id = 1;

-- Отмена транзакции, если денег нет
IF @balance < 1000 THEN
    ROLLBACK;
END IF;

-- Списание и зачисление
UPDATE accounts SET user_balance = user_balance - 1000 WHERE user_id = 1;
UPDATE accounts SET user_balance = user_balance + 1000 WHERE user_id = 2;

-- Применение изменений в базу (фиксация)
COMMIT;
```

### Основные команды:

- START TRANSACTION (или BEGIN) — старт.
    
- COMMIT — фиксирует изменения, делает их постоянными и снимает блокировки.
    
- ROLLBACK — отменяет все действия до состояния перед началом транзакции.
    

### Точки сохранения (Savepoints)

Позволяют откатиться не в самое начало, а к промежуточной точке транзакции.

code SQL

downloadcontent_copy

expand_less

```
START TRANSACTION;

-- Создаем точку сохранения
SAVEPOINT before_updating_user_1;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 1;

-- Откат только до этой точки (оставит транзакцию открытой, но отменит UPDATE выше)
ROLLBACK TO SAVEPOINT before_updating_user_1;

UPDATE accounts SET balance = balance + 200 WHERE user_id = 2;

COMMIT;
```

**Важно помнить:**

1. SAVEPOINT не сохраняет данные в базу навсегда. Для постоянного сохранения всё равно нужен COMMIT.
    
2. Если вызовете просто ROLLBACK (без имени точки), все точки сохранения сгорят, а транзакция откатится полностью.
